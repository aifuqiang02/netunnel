# Netunnel 产品推广执行方案

## 目标

- 让目标用户先知道 `Netunnel` 是什么
- 让一部分目标用户在 5 分钟内真实跑通第一次使用
- 在冷启动阶段优先获取真实试用与反馈，而不是单纯追求曝光量

## 当前产品定位

基于当前根目录 `README.md`，`Netunnel` 的对外定位是：

- 内网穿透，`0.5 元/GB，1 元起用`
- 支持把本地网站、本地 API、本地 TCP 服务发布到公网
- 支持域名入口
- 支持通过桌面客户端统一管理设备、隧道和访问地址

当前更适合对外强调的不是技术实现，而是用户结果：

- 几分钟把本地服务发布到公网
- 让开发联调、临时演示、远程访问这类场景更简单

## 优先目标用户

冷启动阶段不要同时打太宽，先聚焦最容易转化的人群。

### 第一优先级：开发者

适用场景：

- 本地 API 给前端、测试或第三方联调
- 微信、支付、Webhook 回调本地调试
- 临时给客户或同事演示本地页面

优先选择这类人群的原因：

- 需求更高频
- 更容易理解产品价值
- 更愿意尝试新工具
- 更容易在技术社区形成传播

### 第二优先级：个人技术用户

适用场景：

- 远程访问家里电脑
- 远程访问 NAS
- 暴露 Windows 远程桌面等 TCP 服务

### 第三优先级：小团队

适用场景：

- 给客户临时演示本地服务
- 给内部成员开放临时访问入口

## 核心推广原则

### 先卖结果，不先卖技术

用户更关心：

- 能不能把本地服务发到公网
- 上手快不快
- 成本高不高
- 会不会很麻烦

因此推广内容优先讲：

- 场景
- 结果
- 时间成本
- 使用门槛

不要一上来就讲：

- bridge
- agent
- data session
- 架构设计

### 先让用户跑通一次，再谈长期留存

对 `Netunnel` 这类产品，第一次成功体验比品牌曝光更重要。

推广链路应围绕下面五步设计：

1. 用户看到产品
2. 用户 10 秒内看懂产品用途
3. 用户 3 分钟内知道怎么开始
4. 用户 5 分钟内跑通第一次穿透
5. 用户愿意继续使用或反馈问题

### 冷启动阶段优先做内容和试用，不优先买量

现阶段更适合：

- 场景内容分发
- 技术社区答疑
- 小范围试用招募
- 种子用户反馈

不适合优先做：

- 大规模广告投放
- 过早做泛品牌宣传

## 对外统一表达

建议先固定一版一句话卖点，后续在首页、README、视频文案、社区发帖里统一使用。

可选版本：

### 版本 A

`更便宜、更直接的内网穿透。登录桌面客户端，几分钟把本地服务发布到公网。`

### 版本 B

`给开发者用的内网穿透：本地 Web/API/TCP 服务，几分钟发布到公网，按量付费，1 元起用。`

### 版本 C

`本地服务公网访问更简单：支持 HTTP、TCP、域名隧道，桌面端统一管理，按量计费。`

使用要求：

- 首页、README、渠道简介尽量统一表述
- 不同渠道可以调整长短，但核心卖点不要频繁变动

## 必须先补齐的推广资产

在扩大曝光前，先把用户点击进来之后最关键的转化资产补齐。

### 1. 首页或 README 首屏文案

首页至少要回答 4 个问题：

- 这是什么产品
- 能做什么
- 适合谁
- 怎么开始

建议首页结构：

1. 第一屏：一句话价值 + 下载按钮 + 教程入口
2. 第二屏：适用场景
3. 第三屏：为什么选 `Netunnel`
4. 第四屏：3 步上手
5. 第五屏：真实截图或录屏

### 2. 60 秒演示视频

视频目标不是讲原理，而是快速让用户看到结果。

建议流程：

1. 本地启动一个服务
2. 打开 `Netunnel`
3. 创建隧道
4. 获取公网地址
5. 用另一台设备或浏览器访问成功

这个视频可以重复用于：

- 官网
- B 站
- 抖音
- 视频号
- 知乎回答
- 小红书

### 3. 5 分钟上手教程

教程应围绕一个具体任务编写，而不是泛讲产品。

建议优先写这个主题：

- `5 分钟把本地 Web 或 API 服务发布到公网`

教程结构建议：

1. 谁适合看
2. 下载客户端
3. 登录与选择设备
4. 创建一个 HTTP 隧道
5. 验证公网访问成功
6. 常见问题

### 4. 首次体验激励

建议给新用户一个明确可感知的试用理由，例如：

- 新用户送一定额度的流量或余额
- 第一次创建成功隧道后再送一次奖励
- 邀请体验后双方获得少量奖励

重点不是优惠力度，而是推动用户完成第一次真实体验。

## 内容选题方向

不要只写“我们做了一个内网穿透产品”，而要围绕用户正在搜索的问题写内容。

建议优先做以下主题：

1. 本地接口怎么给前端联调
2. 微信支付回调本地怎么调试
3. 企业微信、飞书、Webhook 本地怎么接收
4. 没有公网 IP 怎么访问家里电脑
5. 本地网站怎么临时给客户演示
6. Windows 远程桌面怎么映射到公网
7. HTTP 隧道和 TCP 隧道有什么区别
8. `frp`、`cpolar`、`natapp`、`ngrok` 这类方案怎么选

这些内容的目标不是单纯写教程，而是把用户的问题自然引到 `Netunnel` 的解决方案上。

## 各渠道执行方式

### GitHub / README

把仓库首页当作公开销售页来优化，不只当源码说明。

建议补充：

- 下载入口
- 使用场景
- 3 步上手
- 计费说明
- 常见问题
- 用户交流群

### B 站

建议优先发 3 类视频：

- 1 分钟效果演示
- 3 到 5 分钟场景教程
- 方案对比类内容

标题尽量偏搜索，不要写得过虚。

示例标题：

- `本地接口如何暴露公网？3 分钟搞定`
- `微信支付回调本地调试，内网穿透实战`
- `没有公网 IP，如何远程访问家里电脑`

### 知乎

优先去回答已有问题，而不是只发品牌文章。

适合关注的话题：

- 内网穿透
- 本地接口公网访问
- 微信回调调试
- `frp` 替代
- `ngrok` 替代
- NAS 远程访问

回答方式：

- 先解决问题
- 再自然提到 `Netunnel`
- 不要一上来硬广

### 掘金 / CSDN / 博客园

优先发布教程型内容，做场景搜索覆盖。

示例标题：

- `内网穿透实战：把本地 Web 服务发布到公网`
- `开发调试必备：本地 webhook 回调接收方案`
- `没有公网 IP，如何访问本地 API`

### QQ 群 / 技术社群

不建议直接群发广告。

更有效的方式：

- 先回答群友问题
- 在对方有明确需求时再贴产品
- 带上截图、价格说明或体验入口

## 种子用户策略

不要先追求大量用户，先找 20 个真实愿意试的人。

目标用户来源：

- 你认识的开发者朋友
- 技术群里的尝鲜用户
- 对类似工具有使用习惯的人
- 在社区里明确提过相关需求的人

招募时不要说“支持一下”，而要说“帮我试一个具体场景”。

推荐话术：

`我做了个内网穿透工具，想找 20 个开发者试用。如果你平时有本地接口联调、回调调试、临时给别人看页面的需求，我送你一点体验额度，你帮我跑通一次，然后告诉我哪里卡。`

关注重点：

- 用户卡在哪一步
- 用户哪一步不理解
- 用户是否能在没有额外解释时完成第一次穿透

## 首用体验优化重点

推广前要重点检查下面这些首用节点是否顺畅：

1. 下载后是否容易安装
2. 登录是否顺利
3. 添加设备是否好理解
4. 创建隧道时字段是否过多
5. 创建完成后是否知道怎么验证成功
6. 出错提示是否足够直白

优先优化项：

- 首次打开后的引导
- 一个清晰的新手任务入口
- 成功后突出显示访问地址
- 可直接复制访问地址
- 常见错误给出具体下一步操作

如果后续要做功能优化，可以优先考虑：

- `创建第一个测试 HTTP 隧道` 的新手入口

## 30 天执行节奏

### 第 1 周：补转化基础

1. 统一一句话卖点
2. 重写首页或 README 首屏文案
3. 准备下载入口
4. 准备 5 分钟上手教程
5. 录 60 秒演示视频

### 第 2 周：开始内容投放

1. 发布 1 个 B 站演示视频
2. 发布 2 篇教程文章
3. 回答 10 个相关社区问题
4. 在技术群里做小范围试用招募

### 第 3 周：获取种子用户反馈

1. 找到 20 个试用用户
2. 记录每个用户的卡点
3. 修复前 3 个最影响转化的问题
4. 收集 3 条真实反馈

### 第 4 周：放大有效动作

1. 把真实反馈整理成案例
2. 再发布 2 到 3 篇场景内容
3. 上线邀请或试用激励
4. 持续围绕搜索型问题输出内容

## 早期指标

早期阶段不要只看曝光，优先看这些数据：

1. 下载页访问人数
2. 下载转安装率
3. 安装转登录率
4. 登录转首个隧道创建率
5. 首个隧道创建成功率
6. 次日留存
7. 7 日留存
8. 付费转化率
9. 用户最常见失败原因

核心判断标准：

- 不是有多少人看到了产品
- 而是有多少人第一次真的用成功了

## 当前建议的第一步

先不要分散做很多事，第一步只做一件事：

### 重写产品首页或根目录 README 的首屏文案

原因：

- 这是所有推广动作的入口
- 以后发视频、发文章、发社群时，用户都会回到这里看
- 如果这里讲不清楚，后面的流量会被浪费

这一版文案至少要包含：

1. 一句标题，讲清核心价值
2. 一句副标题，讲清适用场景和特点
3. 一个下载入口
4. 一个 3 步上手说明
5. 一个适用人群说明

建议你处理顺序如下：

1. 先确定一句话卖点
2. 再确定首页标题和副标题
3. 再补 3 步上手
4. 最后补下载入口和体验引导

如果你准备按这个顺序推进，下一步最适合继续做的是：

- 先把 `README.md` 的首页文案改成更像销售页的版本

## 协作方式

接下来可以按“一次只处理一项”的方式推进，例如：

1. 先定一句话卖点
2. 再改 README 首屏
3. 再做 5 分钟教程
4. 再列 20 个内容选题
5. 再做种子用户招募话术

这样推进更容易落地，也更容易逐步看到效果。
